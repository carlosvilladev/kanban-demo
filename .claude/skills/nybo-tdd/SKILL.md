# nybo-tdd

Tests are specifications, not validation. A test written before any implementation
exists forces you to articulate exactly what behavior you intend — in executable form.
Implementation is the act of making that specification pass.

This skill governs how you carry out every implementation task. It does not replace
the plan — it disciplines how the plan is executed, task by task.

---

## 1. Load context

- Read the spec from **`docs/<n>/spec/spec.md`** where `<n>` is the feature name.
- Read the TDD contracts guide from **`docs/<n>/contract/contracts.md`**.
- Read **`docs/<n>/contract/contracts-status.yaml`** (if present) to know each behavior's current TDD phase.
- Read **`.nybo/memory/CORE.md`** and any relevant domain memory from **`.nybo/memory/domains/`**.
- Identify the test framework in use (from CORE.md stack — e.g. Vitest, Jest, pytest) and the exact command to run the suite.
- Check whether a `testing-conventions.md` or testing domain file exists in **`.nybo/memory/domains/`**. If it does, read it before cycle 1. Apply established patterns — don't reinvent what was already decided.

## 2. Identify the next behavior

- Scan `contracts-status.yaml` for the next behavior with `phase: todo`.
- If the user specifies a behavior name, use that instead.
- If no `todo` behaviors remain, report completion and stop.
- Announce which behavior you will implement next.

## 3. Phase: Red — Write one failing test

Convert the `it.todo('...')` in the contract file into a real `it()` with assertions that express the expected behavior.

**What to do:**
- Write the test with real imports and real assertions. The imports may reference modules that don't exist yet — that's expected.
- Run the suite. Capture the output.
- Confirm the test fails **for the right reason** — it should fail because the behavior doesn't exist yet (missing module, missing function, wrong return value), not because of a syntax error or unrelated import.
- Update **`contracts-status.yaml`** → set this behavior's phase to `red`.

**Report to the human:**
\`\`\`
🔴 RED
Test:   <test name or description>
File:   <test file>:<line>
Fails because: <what doesn't exist yet — one sentence>

<runner output showing the failure>
\`\`\`

**Gate:** Do not write implementation code until the human has seen this output. The failing test is the evidence that TDD is being applied genuinely. No Red output means no TDD — it means the implementation came first.

---

## 4. Phase: Green — Write minimum implementation

Write the minimum code that makes the failing test pass.

**Constraints:**
- Write only what is demanded by this specific test. Resist adding logic for cases not yet tested — that comes in future cycles.
- "Minimum" means the simplest thing that works, even if it looks naive. The refactor phase is for improvement.
- Enforce the file size limit from `.nybo/foundation/conventions.yaml`. If a file would exceed the limit, split into focused modules.

**Run the suite. Capture the output.**

**Report to the human:**
\`\`\`
🟢 GREEN
Changed: <file>:<line range> — <what was added, one line>

<runner output showing the test now passes>

Suite: <N passing> / <N total> — <N failing if any>
\`\`\`

- Update **`contracts-status.yaml`** → set this behavior's phase to `green`.

**Gate:** Human reviews and confirms the cycle can advance.

---

## 5. Phase: Refactor — Improve without changing behavior

With all tests green, improve the code without changing its behavior.

**What to look for:**
- Duplication that emerged from the Green phase.
- Names that became clearer once the behavior is visible.
- Abstractions that are now warranted (but weren't before).
- Convention violations flagged by lint or the project's conventions.

**Constraint:** Run the full suite after every refactor step. If anything goes red, undo the refactor — behavior changed.

**Report to the human:**
\`\`\`
🔵 REFACTOR
What changed: <brief description of structural improvement>
Suite: <N passing> / <N total> — all green
\`\`\`

- Update **`contracts-status.yaml`** → set this behavior's phase to `refactored`.

If no refactoring is needed, report:
\`\`\`
🔵 REFACTOR — no changes needed, code is clean
\`\`\`

---

## 6. Cycle complete — repeat or stop

After one complete Red → Green → Refactor cycle:

- Show the contracts-status summary:
\`\`\`
Contracts: ⬜ <todo> 🔴 <red> 🟢 <green> 🔵 <refactored> — <total> total
\`\`\`
- Ask: **"Continue with the next behavior, or stop here?"**

Each cycle implements **one behavior** — not a class, not a feature, one behavior.

If you cannot identify the next behavior to test, stop and ask the human. That is a planning gap, not an execution gap.

---

## Verification Protocol

The human verification gate exists for a specific reason: agents can produce tests that pass from the start, either by writing implementation before the test, or by writing a test that cannot fail. The Red phase output is the only observable evidence that the cycle was genuine.

**The human does not approve the next cycle without:**
1. The Red phase output — the failing test and runner output
2. The Green phase output — the passing test and suite status
3. The Refactor summary

If you cannot produce Red phase output, the cycle is invalid. Stop. Discard the implementation. Restart from a genuine failing test.

## 7. After each completed TDD session

After each full Red→Green→Refactor cycle, check whether a **testing pattern** emerged that is not already in `.nybo/memory/domains/testing.md` (read it if needed).

If a new pattern was found, append it to **`docs/<feature>/suggestions.md`** (or `docs/<feature>/suggestions.md` depending on project structure) under a `## Testing Patterns` section (create the section if absent):

\`\`\`
## Testing Patterns

[TP-001] <One-line description of the pattern>
Example: <test file path or function name that demonstrates it>
Why: <why it's worth capturing — prevents what mistake or inconsistency?>
\`\`\`

Use `[TP-NNN]` IDs, sequential within the feature. Keep each entry under 3 lines. Do not add entries that duplicate what is already in `testing.md`.

This pre-populates curation fodder. `nybo-curate` will pick up these entries from `suggestions.md` during the next curation run.

## Trust Level Behavior

Read the current trust level from **`.nybo/nybo.config.yaml`** (field `trust.level`) or **`.nybo/trust.yaml`** (field `level`). Adjust checkpoint frequency based on the level:

### 3-tier trust model

| Level | Slug | Display | Behavior |
|---|---|---|---|
| L1 | supervised | Supervisado | Pause after Red and after Green. Auto-proceed through Refactor if tests stay green. |
| L2 | semi-autonomous | Semi-autónomo | Pause only after full cycle (Red+Green+Refactor). Human reviews outcome. |
| L3 | autonomous | Autónomo | Auto-proceed through all phases. Only pause if tests fail unexpectedly. |

If the trust level is not set or unrecognised, default to **L1 supervised** behavior (most conservative). Legacy slugs (`observer`, `collaborator` → L1; `architect` → L2) are coerced on read.
