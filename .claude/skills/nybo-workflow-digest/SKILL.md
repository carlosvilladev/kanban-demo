# nybo-workflow-digest

Generate a per-phase timing digest for a completed feature lifecycle. Reads
`.nybo/events.jsonl` (project-local, agent-agnostic) — no token data, no
session data.

**Usage:** `/nybo-workflow-digest <feature>`

**Not for:** token economics (use `/nybo-insights`) or session continuity (use
`/nybo-session-summary`).

---

## Phase boundary map

| Phase   | Start event              | End event          |
|---------|--------------------------|--------------------|
| Plan    | `spec_created`           | `spec_approved`    |
| Run     | `spec_execution_started` | `verify_passed`    |
| Verify  | `verify_passed`          | `curate_run`       |
| Curate  | `curate_run`             | `pr_created`       |
| Ship    | `pr_created`             | `spec_shipped`     |

**Boundary-ownership rule (half-open intervals):** Each phase window is
`[start_ts, end_ts)`. A boundary event belongs to EXACTLY ONE phase: the
event that OPENS a phase is counted in that phase; the event that CLOSES a
phase belongs to the NEXT phase's open interval. Concretely, `verify_passed`
opens the Verify phase — it is counted in Verify's event count, NOT in Run's.

**Multiple-occurrence rule:** when a boundary event appears more than once for
the same feature (e.g., two `verify_passed` events in a re-verify scenario),
use the LAST occurrence as the phase start/end boundary for elapsed
computation. Count ALL occurrences within the resolved window for event-count.

## Step 1 — Parse events.jsonl

1. Read `.nybo/events.jsonl` line by line. Skip malformed lines silently.
2. Filter to events where `event.spec` OR `event.details.feature` matches
   `<feature>`.
3. For each event, resolve the timestamp:
   - Read `event.timestamp` first.
   - If absent or null, fall back to `event.ts` and mark with `*` provenance.
   - If both are absent or null, skip the event entirely.
4. Sort the retained events chronologically by resolved timestamp.
5. Count total retained events. If < 3, note sparse stream.

**Timestamp fallback notation:** a `*` suffix on a time value means "derived
from legacy `ts` field — not from `timestamp`." This covers real events such
as `plan_critiqued` entries that use `ts` instead of `timestamp`.

---

## Step 2 — Resolve phase boundaries

For each phase, find its start-event and end-event in the retained event list:

- If a boundary event type appears multiple times, take the LAST occurrence.
- Record: `start_ts`, `end_ts`, and the resolved timestamps.
- If start or end is absent, mark the boundary as missing (`—`).

**Elapsed computation:**
- When both boundaries are present: compute delta as a human-readable string:
  `Xm Ys` for minutes+seconds, `Xh Ym` for hours+minutes, `Xd Yh` for
  days+hours.
- When either boundary is missing: elapsed = `—`.

**Event count per phase:**
- Count events whose resolved timestamp satisfies `start_ts <= ts < end_ts`
  (half-open, start-inclusive, end-exclusive) against the phase window.
- Boundary event at `start_ts` is included (it opens the phase).
- Boundary event at `end_ts` is excluded (it opens the next phase).
- When boundaries are missing: event count = `—`.

**Actor (dominant):**
- From events in the phase window, find the plurality `actor` value.
- When no events: `—`.

---

## Step 3 — Cross-reference status.yaml

Read `docs/<feature>/status.yaml` if it exists. Use as fallbacks when event
data is missing:

- `created_at` → Plan start fallback if `spec_created` is absent
- `pr_url` / `pr_number` → Ship outcome if `pr_created` event is absent
- `shipped_at` → Ship end fallback if `spec_shipped` is absent

When a value comes from `status.yaml`, note `(status.yaml)` in the output.

## Step 4 — Render the digest

Output a markdown document to the conversation with the following structure:

### Header block

```
# Workflow Digest — <feature>

**Status:** <from status.yaml or last known event>
**Total elapsed:** <sum of known phase elapseds, or —>
**PR:** <pr_url or pr_number or —>

> Elapsed times are gate-to-gate and include human wait time — not compute time.
```

### Phase table

```markdown
| Phase  | Elapsed (gate-to-gate) | % of total | Event count | Actor | Outcome/artifact |
|--------|------------------------|-----------|-------------|-------|-----------------|
| Plan   | ...                    | ...       | ...         | ...   | ...             |
| Run    | ...                    | ...       | ...         | ...   | ...             |
| Verify | ...                    | ...       | ...         | ...   | ...             |
| Curate | ...                    | ...       | ...         | ...   | ...             |
| Ship   | ...                    | ...       | ...         | ...   | ...             |
```

**Column values:**
- **Elapsed:** gate-to-gate delta per phase, or `—` if boundary missing.
- **% of total:** phase elapsed / total known elapsed × 100, or `—`.
- **Event count:** count of events in `[start_ts, end_ts)` window, or `—`.
- **Actor:** plurality actor from phase window, or `—`.
- **Outcome/artifact:**
  - Plan: `plan_critiqued.details.verdict` if present, else `spec_approved`
    confirmation, else `—`.
  - Run: count of `task_completed` events in window, or `—`.
  - Verify: `verify_passed.details.test_count` or `—`.
  - Curate: `curate_run.details.conventions_added` +
    `curate_run.details.domain_patches` or `—`.
  - Ship: `pr_created.details.pr_url` or `spec_shipped.details.pr_number`,
    falling back to `status.yaml` `pr_url`/`pr_number` with `(status.yaml)`
    note, or `—`.

---

## Step 5 — Data-quality notice

After the table, check for missing boundaries:

- If **any phase** is missing a start or end boundary event, append:
  ```
  > **Data quality:** The following phases have incomplete timing data —
  > boundary events were not emitted:
  > - <Phase>: missing `<event_type>` ...
  ```

- If **total retained event count < 3**, prepend:
  ```
  > **Warning:** Sparse event stream — most phase timings will be unavailable.
  ```

- Use `—` (em dash) consistently as the missing-data sentinel throughout.

---

## Step 6 — Emit advisory event

After rendering the digest, emit one advisory event:

```bash
nybo events log workflow_digest_run --spec <feature> --details '{"phases_complete": N, "phases_missing": M}'
```

Where `N` = count of phases with both boundaries present, `M` = count of
phases with at least one missing boundary.

This event is advisory-only — local observability, NOT mapped to Hub.
