# nybo-design

Convert a design source into a verifiable contract, then converge the implementation by measurement — never by visual impression. **The LLM is the workforce, the script is the gate, the design is the source of truth.**

**Usage:** `/nybo-design <feature>`

---

## 0. State detection

Check for `docs/<feature>/design/design.spec.json`:

- **If design contract missing → extract phase** (section 1).
- **If design contract exists → loop phase** (section 2).

One command, two phases, never one continuous run: the human approval gate between them is non-negotiable. A contract change always re-enters the extract phase through human approval.

---

## 1. Extract phase

Produce THREE artifacts under `docs/<feature>/design/` and stop. **Never implement UI in this phase.**

| Artifact | Audience | Role |
|---|---|---|
| `DESIGN.md` | human + agents | Narrative contract: tokens, hierarchy, hard rules, curated notes |
| `design.spec.json` | gate scripts | Machine contract: verifiable values per node |
| `reference.png` | visual gate | 1x export of the frame (or the original image) |

### 1.1 Detect the source

- URL containing `figma.com` with a `node-id` → Figma route (confidence `exact`).
- Image file (png/jpg/screenshot) → image route (confidence `estimated`).
- Neither available → stop and ask the user for one; never invent values.

### 1.2 Figma route (high fidelity)

Mandatory call order on the Figma MCP server:

1. `get_design_context` with the node link. If truncated (large files), call `get_metadata` for the node map and re-request only the needed nodes.
2. `get_variable_defs` → populate the DESIGN.md token table.
3. `get_screenshot` of the frame → save as `docs/<feature>/design/reference.png`.
4. If Code Connect is configured, `get_code_connect_map` → record node→component mappings (the implementer MUST reuse them).

**Fallback without MCP** (rate limit or server unavailable): Figma REST API. Read the token from the `FIGMA_TOKEN` environment variable or the gitignored file `.nybo/state/figma-token`; never commit it, never echo it into artifacts.

```bash
curl -sH "X-Figma-Token: $FIGMA_TOKEN" "https://api.figma.com/v1/files/<FILE_KEY>/nodes?ids=<NODE_ID>"
curl -sH "X-Figma-Token: $FIGMA_TOKEN" "https://api.figma.com/v1/images/<FILE_KEY>?ids=<NODE_ID>&format=png&scale=1"
```

Golden rule: **copy exact payload values — never round, never infer.** If Figma says `lineHeight: 32`, the spec says `"32px"`.

### 1.3 Image route (bounded fidelity)

1. Ask the user the real frame width (if unanswered, assume 1440 and record the assumption in DESIGN.md).
2. Estimate measurements proportionally from the image dimensions; sample colors; approximate typography visually.
3. Mark ALL nodes `"confidence": "estimated"` (the gate applies ±3px instead of ±1px).
4. Be explicit with the user: **pixel-perfect is not promised from a flat image** — the loop converges to ±3px numeric + the visual gate. For ±1px, they need the Figma file.

### 1.4 Build the artifacts

1. Write `DESIGN.md` from `references/design-md-template.md`.
2. Write `design.spec.json` per `references/design-spec-guide.md`. Non-negotiables from the guide: stable kebab-case node ids (each id IS the `data-design-id` contract), relational `gaps` over absolute positions, `children` counts against structural hallucination, resolved values (not token names), and **container paddings/nav offsets asserted explicitly** — the miss class only the visual gate catches late.
3. Save `reference.png` at the same viewport/scale as the spec (1x export).

Issue all `Write` tool calls for these files in a single message (parallel tool use).

### 1.5 Stop for approval

Present DESIGN.md and design.spec.json to the user. End with exactly:

> Design contract written for `<feature>`. Review the contract (`docs/<feature>/design/`); when it looks right, run `/nybo-design <feature>` again to start the fidelity loop. Nothing has been implemented yet.

Do not start the loop automatically. The contract approval is the human gate.

## 2. Loop phase

### 2.1 Preflight

1. Confirm the dev-server URL with the user (or detect: `localhost:3000`, `localhost:5173`, …). The server must be running.
2. Dependencies in the TARGET repo are consent-gated. Ask exactly:

> Install playwright (+ chromium), pixelmatch and pngjs as devDependencies in this project? (yes/no)

- **yes** → run the single install block below, record the consent in your reply.
- **no** → stop cleanly: the loop cannot measure without them; nothing is installed.

```bash
npm ls playwright >/dev/null 2>&1 || (npm i -D playwright && npx playwright install chromium)
npm ls pixelmatch >/dev/null 2>&1 || npm i -D pixelmatch pngjs
```

### 2.2 Initial implementation (if the UI does not exist yet)

Dispatch the `design-implementer` agent with EXACTLY this context: `docs/<feature>/design/DESIGN.md`, `docs/<feature>/design/design.spec.json`, and the target component/page files. Nothing else — clean context produces better fidelity.

On return, grep-verify that **every `node.id` in the spec exists as `data-design-id` in the code**. Any missing id goes back to the agent before measuring.

### 2.3 Measurement loop (maximum 8 iterations)

For each iteration N (starting at 1):

```bash
node .nybo/workflows/nybo-design/references/measure.mjs --url <DEV_URL> --spec docs/<feature>/design/design.spec.json --out docs/<feature>/evidence/delta-report.json --log docs/<feature>/evidence/fidelity-log.jsonl --iteration N
```

- **Exit 0 (PASS)** → go to 2.4.
- **Exit 1 (FAIL)** → read `docs/<feature>/evidence/delta-report.json` and dispatch the correction to `design-implementer` passing ONLY the `failures` array plus this instruction: *"Apply exactly these numeric corrections. Touch only the listed nodes. Redesign nothing."* Each failure already carries an exact `fix`.
- **Exit 2** → environment problem (server down, invalid spec). Fix and retry the SAME iteration — exit 2 never consumes an iteration.

**Oscillation rule:** check `docs/<feature>/evidence/fidelity-log.jsonl`. If the same property alternates its delta sign across consecutive iterations (fix margin → breaks padding → fix padding → breaks margin), constraints conflict: STOP the loop, split the node into smaller sub-nodes in the spec (a human-approved contract change), or surface the conflict to the user. Do not burn iterations.

**Hard cap:** after 8 iterations without convergence, stop and report the remaining deltas with their history. A component that does not converge in 8 is mis-partitioned.

Report per iteration: checks run, failures, corrected deltas.

### 2.4 Final visual gate (only after numeric PASS)

```bash
node .nybo/workflows/nybo-design/references/visual-diff.mjs --url <DEV_URL> --ref docs/<feature>/design/reference.png --spec docs/<feature>/design/design.spec.json --out docs/<feature>/evidence/visual-diff.png --max-ratio 0.01
```

On failure, inspect `docs/<feature>/evidence/visual-diff.png`: what numbers miss is usually shadows, gradients, border colors, image assets, antialiasing — or an unasserted container padding. Apply exact corrections and return to 2.3 (a visual fix can break a number). If the reference is missing or its dimensions mismatch, report it and close on the numeric gate — never invent a reference.

**Never loosen tolerances to force a pass.** If ±1px is unreachable (subpixel rendering, browser line-height), report why; do not hide it.

### 2.5 Curate and close

1. If the same delta class appeared in 2+ iterations, append the lesson to the **"Curated notes"** section of `DESIGN.md` (e.g., a design-system token resolving to an unexpected value).
2. Evidence stays in `docs/<feature>/evidence/`: `delta-report.json`, `fidelity-log.jsonl`, `visual-diff.png`, plus a final screenshot.
3. Log the gate outcome (advisory event — local observability only, intentionally NOT Hub-mapped):

```bash
# PASS:  {"gate":"design-fidelity","feature":"<feature>","checks":N,"failed":0,"visual_ratio":R,"iterations":N}
nybo events log gate_passed --spec <feature> --details '{"gate":"design-fidelity","checks":138,"failed":0,"visual_ratio":0.0017,"iterations":2}'
# FAIL (cap or oscillation halt): same shape with "failed">0 and a "halt_reason" field
nybo events log gate_failed --spec <feature> --details '{"gate":"design-fidelity","checks":138,"failed":5,"visual_ratio":0.032,"iterations":8,"halt_reason":"cap"}'
```

### 2.6 Stop message

End with exactly this block (substitute `<feature>`):

> Fidelity loop complete for `<feature>` — see `docs/<feature>/evidence/` for the delta report and visual diff. Next step:
> - Direct: `/nybo-verify <feature>` — the verify phase re-runs the gates once as evidence (Step 1.5) and presents the human checklist.
>
> Do not run /nybo-verify automatically. The user picks the path.

## Pipeline stages (extension points)

The loop phase is a staged pipeline. Stage C is reserved for the `nybo-design-flow` follow-up (UX flow gates — navigation graph, click-depth, reachability):

- **Stage A** — numeric gate (`measure.mjs`), per screen
- **Stage B** — visual gate (`visual-diff.mjs`)
- **Stage C** — *(reserved)* flow gate (`flow.mjs`), shares the `data-design-id` rail

## Hard rules

1. Fixed viewport from the spec, always. A different viewport invalidates every measurement.
2. Fonts and animations are handled by the scripts — never measure "by hand" with screenshots.
3. The implementer never edits the contract; the orchestrator never implements. Contract changes re-enter the extract phase.
4. Scripts run from `.nybo/workflows/nybo-design/references/` only — adapter copies are not an execution path.
5. This skill never advances the spec lifecycle and never opens PRs — `/nybo-verify` and `/nybo-ship` own those.

## Trust behavior

| Level | Behavior |
|---|---|
| supervised | Confirm before each correction dispatch; show every delta report. |
| semi-autonomous | Run the loop autonomously; pause only at the 2.6 stop (or on oscillation/cap halt). |
| autonomous | Same as semi-autonomous — the contract approval and verify sign-off remain human regardless of level. |

## Boundary with /nybo-plan design

`/nybo-plan design` records narrative design context (references, journeys, acceptance criteria) inside the spec — no contract, no measurement. `/nybo-design` owns measurable pixel fidelity. Use plan-design for surfaces where ±1px does not matter; use this skill when the design IS the product.
